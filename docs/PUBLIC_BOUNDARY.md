# UMC Core 项目边界与安全规范

## 项目定位

UMC Core 是一个**完全开源、可审计、可复现**的边缘AI工程验证框架项目。项目遵循严格的开源边界原则，确保所有内容都可在开源许可下自由使用、修改和分发。

## 开源许可与授权

本项目采用 MIT 许可证，允许用户在遵守许可证条款的前提下自由使用项目代码。详细的许可条款请参阅项目根目录的 [LICENSE](../LICENSE) 文件。

## 内容边界规范

### 允许包含的内容
为了确保项目的开源性和可协作性，UMC Core 项目仅包含以下类型的内容：

1. **公开知识与工具链**
   - 公开文档和标准的使用方法
   - 开源工具链的集成和配置示例
   - 社区认可的最佳实践和工程模式

2. **可复现的技术实现**
   - 完整的可执行脚本和自动化工作流
   - 标准化的证据包模板和验证工具
   - 工程化的评估框架和测试套件

3. **工程方法论文档**
   - 性能指标的定义和测量方法
   - 内存使用模式和优化策略
   - 系统稳定性的评估框架
   - 实验结果的可复现性保障机制

4. **技术验证材料**
   - 环境配置的标准化描述（env.json）
   - 结构化性能指标（results.json）
   - 技术决策记录（report.md）
   - 完整性校验信息（manifest.json）

### 严禁包含的内容
为确保项目的开源合规性和安全性，以下内容**严格禁止**包含在项目中：

1. **知识产权受限材料**
   - 任何模型权重文件（包括但不限于 `.gguf`, `.bin`, `.pt`, `.pth`, `.safetensors`, `.onnx` 等格式）
   - 受版权保护的数据集或评测集
   - 专利保护的技术实现细节

2. **敏感或私有信息**
   - 公司内部的私有参数、校准表或配置
   - 内部系统路径、代号或标识符
   - 可能泄露商业机密或技术细节的任何信息

3. **不实或不可验证的声明**
   - 未经证据支持的"隐藏能力"或"未来路线"声明
   - 不可验证的性能承诺或保证
   - 限制或惩罚第三方使用的条款

4. **安全风险内容**
   - 硬编码的凭证、密钥或访问令牌
   - 可能被恶意利用的系统漏洞信息
   - 违反法律法规或行业规范的内容

## 安全与合规性保障

### 审计机制
项目内置自动化审计机制，确保所有内容符合开源边界规范：

```powershell
# 运行完整性审计
pwsh .\scripts\audit_public.ps1
```

审计脚本执行以下检查：
1. **证据包验证**：检查所有证据包的完整性和格式合规性
2. **敏感信息扫描**：检测可能包含的敏感或私有信息
3. **许可证合规性**：验证所有内容的开源许可兼容性
4. **完整性校验**：确认所有文件的SHA256校验和一致性

### 问题处理流程
如果发现任何可能违反边界规范的内容，请立即执行以下步骤：

1. **立即停止分发**：暂停项目的所有分发和分享活动
2. **问题隔离**：识别并隔离有问题的文件或内容
3. **安全删除**：彻底删除所有包含敏感信息的文件
4. **重新审计**：运行完整的审计流程确认问题已解决
5. **版本管理**：必要时创建新的版本标签，确保用户不会访问到有问题版本

### 贡献者责任
所有项目贡献者都应：
1. **自我审计**：在提交贡献前运行审计脚本检查合规性
2. **内容审查**：确保所有提交内容都符合边界规范
3. **透明度原则**：明确标注任何第三方依赖或许可要求
4. **安全优先**：优先考虑项目的长期安全性和合规性

## 技术实施保障

### 1. 自动化门禁
项目集成自动化门禁系统，在以下关键点执行边界检查：
- **代码提交时**：检查提交内容是否符合边界规范
- **持续集成时**：运行完整的审计测试套件
- **版本发布时**：验证发布包的安全性和合规性

### 2. 完整性校验
所有技术交付物都包含完整性校验机制：
- **SHA256哈希链**：确保文件从创建到分发的完整性
- **数字签名**：可选的支持，用于验证发布来源
- **版本绑定**：证据包与代码版本严格绑定，防止篡改

### 3. 透明度报告
项目维护完整的透明度报告：
- **依赖清单**：所有第三方依赖的完整清单和许可证信息
- **构建证明**：可验证的构建过程和环境信息
- **安全声明**：已知安全问题和安全实践说明

## 社区协作准则

### 1. 问题报告
如果发现任何边界违规或安全问题：
1. 通过项目Issue系统私下报告
2. 提供具体的文件路径和问题描述
3. 避免公开讨论可能的安全漏洞细节

### 2. 贡献指南
所有贡献都应：
1. 遵循项目的工程方法论和验证框架
2. 包含完整的可复现证据包
3. 通过自动化审计检查
4. 有明确的技术文档和用例说明

### 3. 使用规范
项目用户可以：
1. 自由使用、修改和分发项目代码
2. 基于项目框架构建自己的工程验证体系
3. 贡献改进和建议，帮助项目持续完善
4. 在遵守许可证的前提下将项目用于商业用途

## 长期维护承诺

UMC Core 项目团队承诺：
1. **持续审计**：定期进行全面的安全性和合规性审计
2. **快速响应**：及时处理发现的安全问题和边界违规
3. **透明沟通**：公开所有重要的安全决策和变更
4. **社区参与**：鼓励社区成员参与项目的安全和合规性建设

通过严格遵守这些边界规范和安全实践，UMC Core 项目旨在建立一个可信、安全、可持续的开源工程验证生态系统。
